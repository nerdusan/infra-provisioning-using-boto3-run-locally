# infra-provisioning-using-boto3-run-locally

A lightweight Python automation tool utilizing `boto3` to programmatically provision an Amazon EC2 instance, apply standard operational tags, block execution until the instance is live, and write its allocated Public IP address to a local configuration file for downstream automation.

---

## Architecture & Configuration

The script is hardcoded with predictable infrastructure defaults targeting standard development environments:

* **Region:** `eu-west-2` (London)
* **AMI:** Ubuntu 24.04 LTS (`ami-0b9932f4918a00c4f`)
* **Instance Type:** `t3.micro`
* **Default Tags Applied:**
    * `Name`: `Automated-Dev-Box`
    * `Environment`: `Development`
    * `ManagedBy`: `Boto3-Script`

---

## Design Choice: CLI Input via `argparse`

To maintain a clean, reusable codebase and adhere to security best practices, **the AWS EC2 Key Pair name is explicitly excluded from the source code.** This project implements **Command-Line Arguments (`argparse`)** to input the key pair dynamically at runtime. This pattern was selected because it forces explicit user intent at the moment of execution and prevents accidental resource provisioning with incorrect keys.

### Alternative Configuration Methods Considered

While `argparse` was selected for this implementation, other valid architectural patterns for injecting runtime configurations include:

1.  **Environment Variables (`os.getenv`):** Sourcing variables like `AWS_EC2_KEY_NAME` directly from the shell environment. This is highly efficient for automated CI/CD pipelines (e.g., Jenkins, GitHub Actions) but less explicit during manual terminal execution.
2.  **External Configuration Files (`configparser` or `pyyaml`):** Reading configurations from a local `config.yaml` or `settings.ini` file. This is ideal for managing large matrices of parameters but introduces extra file-management overhead for a single variable.
3.  **AWS Systems Manager (SSM) Parameter Store:** Fetching configuration values directly from AWS at runtime. This centralizes management across teams but adds API overhead and extra IAM permission requirements to the script's execution role.

## Usage

Execute the script by passing your existing AWS EC2 Key Pair name via the `--key-name` flag:

```bash
python provision_ec2.py --key-name your-aws-key-pair
```

### Output Lifecycle
Upon execution, the script performs an atomic provisioning request, blocks until AWS transitions the instance state to running, and captures the network metadata.
It overwrites a local configuration file (instance_config.txt) in the root directory:

```

#Generated by Auto-Provisioning Script
INSTANCE_ID=i-0123456789abcdef0
PUBLIC_IP=54.212.43.102

````

This clean key-value format allows downstream tools, shell scripts, or local SSH configurations to immediately ingest the new environment's IP address.
